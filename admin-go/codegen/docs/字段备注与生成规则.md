# Codegen 字段备注与生成规则

字段备注是 codegen 的核心输入之一。生成器从 `information_schema.COLUMNS.COLUMN_COMMENT` 读取备注，并据此决定前端标签、Tooltip、枚举常量、字典、搜索控件、外键关联、校验规则的一部分。

数据库迁移文件仍然是真源。新增或调整字段时，优先把备注写准确，再执行 codegen。

## 总体语法

```text
{字段标签}[（Tooltip 提示）][:枚举定义或 dict:{字典类型}]|{指令1}|{指令2}|...
```

解析规则：

- `|` 后面的内容是指令，当前支持 `ref:`、`search:`、`keyword:`、`priority:`、`search-priority:`。
- 主体部分只按第一个英文冒号 `:` 或中文冒号 `：` 拆分。冒号左侧是标签，右侧是枚举或字典。
- 标签里的中文括号 `（）` 或英文括号 `()` 会被提取为 Tooltip，括号前文字作为精简标签。
- 空备注会回退成字段名的 DAO 风格 CamelCase，例如 `order_no` 变成 `OrderNo`。这个兜底只用于防崩，不推荐依赖。
- 中文冒号 `：` 在指令部分和枚举部分都会被自动替换为英文冒号 `:`。
- 中文逗号 `，` 在枚举定义中会被自动替换为英文逗号 `,`。

推荐模板：

```sql
`order_no` varchar(64) NOT NULL COMMENT '订单号|search:eq|keyword:on|priority:100'
`title` varchar(100) NOT NULL COMMENT '标题|keyword:on|priority:95'
`pay_amount` bigint unsigned NOT NULL DEFAULT 0 COMMENT '支付金额（分）'
`status` tinyint unsigned NOT NULL DEFAULT 1 COMMENT '状态:0=关闭,1=开启|search:select'
`gender` tinyint unsigned DEFAULT NULL COMMENT '性别:dict:gender|search:select'
`user_id` bigint unsigned NOT NULL COMMENT '归属用户|ref:system_users.username|search:select'
```

## 字段标签

标签直接用于前端表单、列表列头、详情字段、导出表头等位置。标签要写业务名，不要写数据库名。

| 推荐 | 不推荐 | 原因 |
|------|--------|------|
| `订单号` | `order_no` | 页面展示需要中文业务名 |
| `支付金额（分）` | `金额` | 单位要在备注里说清楚 |
| `归属用户` | `用户ID` | 外键字段页面上展示的是业务对象 |
| `状态:0=关闭,1=开启` | `status` | 枚举需要让 codegen 能解析 |

字段标签如果需要说明单位、范围或业务含义，用 Tooltip 括号，不要把长说明堆进主标签。

```sql
`sort` int NOT NULL DEFAULT 0 COMMENT '排序（升序，数字越小越靠前）'
`quota` bigint NOT NULL DEFAULT 0 COMMENT '额度（分，0 表示不限制）'
```

生成效果：

- `Label` 保留完整备注标签，例如 `额度（分，0 表示不限制）`。
- `ShortLabel` 去掉括号，例如 `额度`。
- `TooltipText` 取括号内文字，例如 `分，0 表示不限制`。

括号匹配规则：

1. 优先匹配中文全角括号 `（）`。
2. 如果没有中文括号，再匹配英文半角括号 `()`。
3. 使用 `LastIndex` 匹配最外层右括号，支持嵌套。
4. 括号前的文字作为 `ShortLabel`，括号内的文字作为 `TooltipText`。

特殊字段覆盖：

- `tenant_id` 的标签强制覆盖为 `租户`（忽略 COMMENT 中的标签）。
- `merchant_id` 的标签强制覆盖为 `商户`（忽略 COMMENT 中的标签）。

## 枚举字段

### 枚举格式

```text
字段标签:值1=显示名1,值2=显示名2
```

示例：

```sql
`status` tinyint unsigned NOT NULL DEFAULT 1 COMMENT '状态:0=关闭,1=开启'
`audit_status` tinyint unsigned NOT NULL DEFAULT 0 COMMENT '审核状态:0=待审核,1=已通过,2=已拒绝'
`type` tinyint unsigned NOT NULL DEFAULT 1 COMMENT '类型:1=普通,2=VIP,3=管理员'
```

### 枚举组件自动推导

枚举数量和字段名共同决定前端组件：

| 条件 | 组件 | 示例 |
|------|------|------|
| `status` 或 `is_*` 字段 + 2 个枚举值 | `Switch` | `状态:0=关闭,1=开启` |
| `status` 或 `is_*` 字段 + 3 个及以上枚举值 | `Radio` | `状态:0=草稿,1=已发布,2=已下架` |
| `status` 字段 + 无枚举值 | `Input`（回退） | `状态` |
| `is_*` 字段 + 无枚举值 | `Switch`（默认 0/1） | `是否激活` |
| 其他字段 + 有枚举值 | `Select` | `类型:1=普通,2=VIP,3=管理员` |
| `type` / `level` / `grade` 字段（无枚举） | `Select` | `类型` |

### 枚举生成效果

- 后端 `internal/consts/{module}.go` 生成 Go 常量。
- 前端列表和详情页按枚举显示 `Tag` 标签，带颜色。
- 前端表单按上述规则使用 Switch/Radio/Select。

### 枚举常量命名规则

常量名格式：`{ModelName}{FieldCamel}{NameIdent}`

- `NameIdent` 优先使用语义化英文标识（见下方映射表）。
- 未命中映射表的标签使用 `V{值}` 兜底，例如 `WorkOrderPriorityV5`。

### 枚举注意事项

- 枚举值和标签里不要使用英文逗号 `,` 或等号 `=`，当前没有转义语法。
- 数值要稳定，不要因为显示顺序变化而重排已有值。
- 字段类型可以是 `tinyint`、`int`、`varchar` 等；实际 TS/Go 类型仍由数据库类型决定。
- 重复枚举值会被自动去重，只保留第一个。

## 字典字段

动态字典格式：

```text
字段标签:dict:{字典类型}
```

示例：

```sql
`gender` tinyint unsigned DEFAULT NULL COMMENT '性别:dict:gender|search:select'
`source` varchar(32) NOT NULL DEFAULT '' COMMENT '来源:dict:user_source|search:select'
```

生成效果：

- 前端从系统字典 API 加载选项。
- 搜索控件默认按 `Select` 生成。
- 字典字段不会同时生成静态枚举常量（`EnumValues` 会被清空，`IsEnum` 设为 false）。

使用要求：

- 正常业务仓库应保留 `system/dict` API。
- 只有在精简仓库做模板验证时，才考虑打开 `codegen.yaml` 的 `allow_missing_dict_module` 兜底。

## 搜索指令

字段备注可以用 `|search:{mode}` 覆盖默认搜索识别。

### 搜索模式

| 指令 | 生成效果 | 常用场景 |
|------|----------|----------|
| `search:off` | 不生成独立搜索控件 | 大文本、图片、密钥 |
| `search:on` | 使用默认智能识别 | 只想显式声明该字段参与识别 |
| `search:eq` | 精确匹配 | 编号、编码、唯一字段 |
| `search:like` | 模糊匹配 | 标题、名称、联系人 |
| `search:range` | 区间搜索 | 时间、金额、数量 |
| `search:select` | 下拉筛选 | 枚举、字典、普通外键 |
| `search:tree` | 树形下拉筛选 | 部门、分类、树形外键 |

### 关键词搜索指令

| 指令 | 生成效果 |
|------|----------|
| `keyword:on` | 加入全局关键词搜索，同时保留独立搜索控件 |
| `keyword:off` | 不加入全局关键词搜索 |
| `keyword:only` | 只加入全局关键词搜索，不生成独立控件 |

### 排序指令

```text
priority:100
search-priority:100
```

两者等价，数值越大搜索控件越靠前。

### 默认搜索优先级

未指定 `priority:` 时，搜索控件按以下优先级排序：

| 字段类型 | 默认优先级 |
|----------|-----------|
| 精确搜索字段（`_no`/`_code`/`_sn`） | 100 |
| 名称类（`title`/`name`/`username`/`nickname`/`real_name`） | 95 |
| 联系方式（`phone`/`mobile`/`email` 及其后缀） | 90 |
| 外键字段 | 82 |
| 枚举/字典字段 | 78 |
| 模糊搜索文本 | 70 |
| 时间字段 | 60 |
| 其他 | 50 |

### 默认搜索识别（启发式规则）

未写 `search:` 指令时，生成器按以下规则自动判断：

| 字段类型 | 自动识别结果 |
|----------|-------------|
| `*_at` 时间字段 | `range`（区间搜索） |
| `*_id` 外键字段（关联表是树形） | `tree`（树形下拉） |
| `*_id` 外键字段（关联表非树形） | `select`（下拉筛选） |
| `parent_id` | `tree`（树形下拉） |
| 枚举字段或字典字段 | `select`（下拉筛选） |
| 可搜索文本 + 精确搜索后缀 | `eq`（精确匹配） |
| 可搜索文本（其他） | `like`（模糊匹配） |
| 隐藏字段、ID、密码字段 | 不生成搜索控件 |

自动模糊搜索字段如果同时参与关键词搜索，列表页最多保留前两个独立模糊控件，避免搜索区过宽。显式 `search:like` 不受这个自动裁剪影响。

### 全局关键词搜索

- 需要至少 2 个以上 `keyword:on` 字段时，才会启用全局关键词搜索框（`HasKeywordSearch=true`）。
- `keyword:only` 的字段只参与全局关键词搜索，不生成独立搜索控件。
- 没有写 `keyword:` 的模糊搜索字段（`SearchOperator=like`），会自动参与关键词搜索。

### 推荐写法

```sql
`order_no` varchar(64) NOT NULL COMMENT '订单号|search:eq|keyword:on|priority:100'
`title` varchar(100) NOT NULL COMMENT '标题|search:like|keyword:on|priority:95'
`summary` varchar(255) NOT NULL DEFAULT '' COMMENT '摘要|keyword:only'
`status` tinyint unsigned NOT NULL DEFAULT 1 COMMENT '状态:0=关闭,1=开启|search:select|priority:80'
`created_at` datetime DEFAULT NULL COMMENT '创建时间|search:range'
```

## 外键指令

### 默认规则

- `*_id` 且为整数类型（`bigint`/`int`/`smallint`/`tinyint`/`mediumint`），会被识别为单选外键。
- `*_ids` 会被识别为多选外键。
- `parent_id` 会被识别为树形父级字段。
- 以下字段不按普通 `*_id` 外键处理：`id`、`dept_id`、`parent_id`。
- `varchar`/`char` 类型的 `*_id` 字段**不会**按真正外键处理（视为业务关联 ID）。
- 密码类字段（如 `*_secret_id`、`*_access_key`）也不会按外键处理。

### 默认外键表推断

优先按"当前应用 + 字段名前缀"推断。例如 `demo_article.category_id` 默认指向 `demo_category`。

推断过程：

1. 去掉 `_id` 后缀得到关联模块名（如 `category_id` → `category`）。
2. 先尝试 `{当前应用}_{模块名}`（如 `demo_category`），检查该表是否存在。
3. 如果不存在，再尝试裸模块名（如 `category`）。
4. 在关联表中按优先级查找显示字段。

### 显示字段查找优先级

```text
title, sku_no, name, username, nickname, real_name, label, phone, mobile, order_no, code, no, email
```

如果按以上优先级都找不到显示字段，codegen 会报错并中止。

### ref 指令语法

跨应用或字段名不明确时，必须用 `ref:` 显式声明：

```sql
`user_id` bigint unsigned NOT NULL COMMENT '作者|ref:system_users.username|search:select'
`tenant_id` bigint unsigned NOT NULL COMMENT '租户|ref:system_tenant.name|search:select'
`merchant_id` bigint unsigned DEFAULT NULL COMMENT '商户|ref:system_merchant.name|search:select'
```

`ref:` 格式：

```text
ref:{关联表}
ref:{关联表}.{显示字段}
```

如果没有指定显示字段，按上述优先级在关联表中自动查找。

### 多外键同表消歧

当多个外键指向同一张关联表时（如 `author_id` 和 `reviewer_id` 都指向 `users`），生成的 `RefFieldName` 会冲突。生成器自动用字段名前缀消歧：

| 字段 | 默认 RefFieldName | 消歧后 |
|------|-------------------|--------|
| `author_id` | `UsersUsername` | `AuthorUsername` |
| `reviewer_id` | `UsersUsername` | `ReviewerUsername` |

如果消歧后仍然冲突（极端情况），该外键会被跳过。

### 特殊外键

`tenant_id` 和 `merchant_id` 有特殊处理：

- 如果 COMMENT 中没有 `ref:` 指令，自动注入 `ref:system_tenant.name` 和 `ref:system_merchant.name`。
- 标签强制覆盖为 `租户` / `商户`。
- 前端列表列和搜索筛选仅对平台超级管理员显示。

### 生成效果

- 后端列表/详情批量查询关联表显示字段，避免 N+1 查询。
- 前端表单导入关联模块列表 API，生成下拉选项。
- 搜索表单对外键生成下拉或树形下拉。
- 关联表是否树形（`RefIsTree`）由是否有 `parent_id` 列自动判断。
- 关联表是否有 `deleted_at`（`RefHasDeletedAt`）用于安全生成软删过滤。
- 关联表是否有 `tenant_id`/`merchant_id` 用于租户隔离查询。

### 注意事项

- 跨应用外键一定要写 `ref:`，否则容易推断到错误应用。
- `parent_id` 的关联表始终指向本表自身（树形结构）。

## 组件映射

字段备注不直接指定组件，组件主要由字段名、类型、枚举和字典共同决定。组件映射逻辑在 `parser/field_mapper.go` 的 `MapComponent` 函数中实现。

### 组件映射优先级

匹配按以下顺序进行，命中即停：

| 优先级 | 规则 | 组件 |
|--------|------|------|
| 1 | `parent_id` | `TreeSelectSingle` |
| 2 | `parent_ids` | `TreeSelectMulti` |
| 3 | `*_ids` 后缀 | `SelectMulti` |

## 字段特征自动检测

以下特征由字段名和数据库类型自动检测，不需要在备注中声明。检测规则定义在 `parser/rule_registry.go` 和 `parser/parser.go` 中。

### 密码/敏感字段（`IsPassword`）

精确名匹配：`password`

后缀匹配：`_password`、`_pwd`、`_secret`、`_secret_key`、`_secret_id`、`_access_key`、`_access_token`、`_refresh_token`、`_api_token`

> **缩窄规则**：普通的 `_token` 后缀**不会**触发密码检测（如 `invite_token`、`device_token`）。只有带 `_access_token`、`_refresh_token`、`_api_token` 前缀的才会。

生成效果：

- 表单使用 `Password` 组件，编辑时允许为空（仅传入非空时加密更新）。
- 列表和详情页不展示。
- 不生成搜索控件。
- 验证规则：`length:6,32`。

### 金额字段（`IsMoney`）

精确名匹配：`price`、`amount`、`balance`、`income_total`、`income_balance`

后缀匹配：`_price`、`_amount`、`_balance`、`_income`、`_fee`、`_cost`、`_deposit`、`_refund`

生成效果：

- 表单使用 `InputNumber` 组件。
- 列表和详情页显示 `÷ 100` 并保留两位小数（分 → 元）。
- 导入/导出按"分"存储。

### 图片字段名（`isImageFieldName`）

精确名匹配：`avatar`、`cover`、`logo`、`banner`、`thumbnail`、`poster`

后缀匹配：`_image`、`_img`、`_photo`、`_pic`、`_cover`、`_banner`、`_logo`、`_thumbnail`、`_poster`、`_avatar`

生成效果：

- 表单使用 `ImageUpload` 组件。
- 列表和详情页渲染 `<img>` 标签（带 `https?://` 协议白名单校验防 XSS）。

### 可搜索文本字段（`IsSearchable`）

精确名匹配：`title`、`name`、`username`、`nickname`、`phone`、`mobile`、`email`、`real_name`、`order_no`、`remark`、`description`、`summary`、`intro`、`address`、`contact`、`contact_name`、`link_url`、`url`、`keyword`

后缀匹配：`_name`、`_title`、`_remark`、`_desc`、`_description`、`_summary`、`_intro`、`_phone`、`_mobile`、`_email`、`_address`、`_keyword`、`_no`

附加条件：Go 类型必须是 `string`，且不是 ID / 外键 / 密码字段。

### 精确搜索字段（`IsExactSearch`）

精确名匹配：`no`、`code`、`sn`

后缀匹配：`_no`、`_code`、`_sn`

这些字段的搜索操作符使用 `=` 而非 `LIKE`。

### 时间字段（`IsTimeField`）

后缀匹配：`_at`

生成效果：

- 搜索使用区间选择器。
- 不参与 CSV 导入/导出。
- 不计入表单字段（隐藏字段除外）。

### 隐藏字段（`IsHidden`）

精确名匹配：`id`、`created_at`、`updated_at`、`deleted_at`、`created_by`、`dept_id`

这些字段不生成表单输入，不参与前端必填校验。

### 外键字段识别条件

同时满足以下条件才被识别为外键（`IsForeignKey=true`）：

1. 字段名以 `_id` 结尾
2. 字段名不是 `id`、`dept_id`、`parent_id`
3. 数据库类型是整数类型（`bigint`/`int`/`smallint`/`tinyint`/`mediumint`）
4. 不是密码/敏感字段

### 必填字段（`IsRequired`）

同时满足以下条件：

1. `IS_NULLABLE = 'NO'`
2. 没有 `DEFAULT` 值
3. 字段名不是 `id`
4. 不是 `auto_increment`
5. 不是隐藏字段（隐藏字段的 `IsRequired` 会被置为 false）

## 校验规则

后端和前端校验主要由数据库结构和字段名推导。校验逻辑在 `parser/parser.go` 的 `buildValidationRules` 函数中实现。

### 后端验证规则（`ValidationRules`）

| 规则来源 | 生成的规则 | 示例 |
|----------|----------|------|
| `NOT NULL` 且无默认值，非主键、非自增、非隐藏 | `required` | `required` |
| `email` 或 `*_email` 字段名 | `email` | `email` |
| `phone`/`mobile`/`*_phone`/`*_mobile` 字段名 | `phone-loose` | `phone-loose` |
| `url`/`*_url`/`*_link` 字段名 | `url` | `url` |
| `varchar(n)`/`char(n)` 类型（GoType=string，有 MaxLength，非密码） | `max-length:{n}` | `max-length:200` |
| `password`/`*_password`/`*_pwd` 等密码字段 | `length:6,32` | `length:6,32` |

### 前端验证规则（`FrontendRules`）

前端验证规则标识用于生成 Vben 表单 `rules` 属性：

| 字段类型 | 必填时 | 非必填时 |
|----------|--------|----------|
| email 类 | `requiredEmail` | `email` |
| phone 类 | `requiredPhone` | `phone` |
| url 类 | `requiredUrl` | `url` |

### 更新接口验证

更新接口会自动去掉 `required` 规则，保留格式、长度等规则。这是因为更新时只更新传入的字段，未传入的字段不应触发必填校验。

密码字段编辑时允许为空，只有传入非空密码才重新加密。

## 隐藏字段与自动字段

以下字段默认不生成表单输入：

```text
id, created_at, updated_at, deleted_at, created_by, dept_id
```

生成行为：

| 字段 | 来源 | 说明 |
|------|------|------|
| `id` | 后端 Snowflake | 自动生成，前端使用 `string` 类型（防精度丢失） |
| `created_at` | 数据库/GoFrame | 自动维护 |
| `updated_at` | 数据库/GoFrame | 自动维护 |
| `deleted_at` | GoFrame 软删 | 自动维护 |
| `created_by` | 当前登录用户 | 后端自动注入，用于数据权限 |
| `dept_id` | 当前登录用户部门 | 后端自动注入，用于数据权限 |

`tenant_id` 和 `merchant_id` 不在通用隐藏字段列表里，因为平台管理员有时需要在平台级页面选择归属。生成器会基于这两个必需字段生成租户/商户权限守卫：

- 租户/商户账号创建和更新时，后端会覆盖请求里的租户/商户归属。
- 平台超级管理员没有租户限制，可以操作平台级和跨租户数据。
- 前端列表列和搜索筛选只对平台超级管理员显示；判断条件来自 `auth/info.isAdmin`，同时要求 `tenantId=0`、`merchantId=0`。
- 导入时租户/商户字段不从 CSV 读取，由当前登录上下文注入。

普通租户、商户或员工账号即使伪造 `tenant_id` / `merchant_id` 查询参数，后端仍会先套租户/商户数据权限，不允许跨主体访问。

## SaaS 与数据权限生成约定

### 租户/商户权限必需字段

```text
表中必须同时存在 tenant_id 和 merchant_id 字段
```

`merchant_id=0` 表示平台级或租户级数据；`merchant_id>0` 表示商户级数据。字段必须存在，不代表每行都必须绑定商户。

### 生成的后端公共方法调用

| 场景 | 生成调用 |
|------|----------|
| 新增 | `ApplyTenantScopeToWrite` + `EnsureTenantMerchantAccessible` |
| 更新 | `ApplyTenantScopeToWrite` + `EnsureTenantMerchantAccessible` + `EnsureTenantScopedRowAccessible` |
| 删除 | `EnsureTenantScopedRowAccessible` |
| 批量删除 | `EnsureTenantScopedRowsAccessible` |
| 详情 | `EnsureTenantScopedRowAccessible` |
| 列表 | `ApplyTenantScopeToModel` |
| 树形 | `ApplyTenantScopeToModel` |
| 导出 | `ApplyTenantScopeToModel` |
| 批量更新 | `EnsureTenantScopedRowsAccessible` |
| 导入 | `ApplyTenantScopeToWrite` + `EnsureTenantMerchantAccessible` |

### 部门数据权限必需字段

```text
表中必须同时存在 created_by 和 dept_id 字段
```

生成行为：

- 新增和导入自动写入 `created_by` / `dept_id`。
- 列表、树形和导出会调用 `ApplyDataScope`。
- 这套部门权限是既有组织权限模型；租户权限用于 SaaS 运营主体隔离，两者可以同时存在。
- 生成器会在解析表结构时校验 `tenant_id`、`merchant_id`、`created_by`、`dept_id` 四个字段，缺字段时不落盘任何 CRUD 文件。

## 表设计检查清单

生成前逐项检查：

- [ ] 表名符合 `{应用名}_{模块名}`，例如 `system_domain`、`upload_file`
- [ ] 每个业务字段都有中文备注，且备注能直接作为页面文案
- [ ] 金额字段备注写清单位，推荐统一用"分"
- [ ] 枚举字段使用 `状态:0=关闭,1=开启` 这种可解析格式
- [ ] 枚举标签使用映射表中已收录的中文（避免 V{n} 兜底）
- [ ] 动态字典使用 `字段:dict:{type}`，不要和静态枚举混用
- [ ] 跨应用外键写 `ref:{table}.{display}`
- [ ] 编号类字段写 `search:eq`，名称/标题类字段按需写 `keyword:on`
- [ ] 大文本、密钥、图片、文件字段按需写 `search:off`
- [ ] 所有 codegen CRUD 表必须同时保留 `tenant_id`、`merchant_id`、`created_by` 和 `dept_id`
- [ ] 密码字段使用 `_password`/`_pwd`/`_secret`/`_secret_key`/`_access_key`/`_access_token`/`_refresh_token`/`_api_token` 后缀
- [ ] 图片字段使用 `_image`/`_img`/`_photo`/`_pic`/`_avatar`/`_logo`/`_cover`/`_banner`/`_thumbnail`/`_poster` 后缀或精确名
- [ ] 金额字段使用 `_price`/`_amount`/`_balance`/`_income`/`_fee`/`_cost`/`_deposit`/`_refund` 后缀或精确名
- [ ] 时间字段使用 `_at` 后缀（如 `published_at`、`expired_at`）
- [ ] URL 字段使用 `_url` 或 `_link` 后缀
- [ ] 富文本字段使用 `content`/`body` 精确名或 `_content`/`_body`/`_html` 后缀
