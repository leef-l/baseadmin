# GBaseAdmin 基础框架说明

当前仓库已经收缩为一套只保留后台基础能力的框架，范围如下：

- `admin-go/app/system`：登录、用户、角色、部门、菜单、权限
- `admin-go/app/upload`：文件、目录、目录规则、上传配置
- `vue-vben-admin/apps/web-antd/src/`：管理端前端，仅保留 `system` / `upload` 对应模块与公共壳
- `admin-go/codegen/`：代码生成器
- `admin-go/database/migrations/`：数据库结构和初始化真源

## 当前保留模块

后端当前实际挂载的核心模块：

- `system`：`auth`、`dept`、`role`、`menu`、`users`，以及公开路由 `health`、`hello`
- `upload`：`config`、`dir`、`dir_rule`、`file`、`uploader`，以及公开路由 `health`

前端当前保留页面和共享能力：

- `views/system/dept`、`views/system/menu`、`views/system/role`、`views/system/users`
- `views/upload/config`、`views/upload/dir`、`views/upload/dir_rule`、`views/upload/file`
- 共享逻辑集中在 `api/core`、`components/file-manager`、`components/upload`、`components/tinymce`、`components/json-editor`

## 管理端说明

- 路由壳仍使用 Vben Admin
- `preferences.app.accessMode = 'backend'`
- `preferences.app.defaultHomePath = '/system/dept'`
- 前端请求基地址是 `/api`
- 开发期 Vite 代理把 `/api/system` 转发到 `127.0.0.1:44003`
- 开发期 Vite 代理把 `/api/upload` 转发到 `127.0.0.1:44004`
- `apps/web-antd/.env.development` 默认使用上述本机代理目标
- Docker 前端容器则通过 `docker/dev/.env` 注入 `VITE_PROXY_SYSTEM_TARGET=http://system:8000`、`VITE_PROXY_UPLOAD_TARGET=http://upload:8002`
- 登录主链路依赖 `/system/auth/login`、`/system/auth/info`、`/system/auth/menus`
- 动态菜单来自后端 `/system/auth/menus`，并在 `src/api/core/menu-transformer.ts` 转成 Vben 路由
- 菜单 `type=4` 会映射成外链 `meta.link`
- 菜单 `type=5` 会映射成内嵌页 `meta.iframeSrc`
- 外链 / 内嵌页在后端未填写 `component` 时，前端会兜底为 `IFrameView`
- `src/router/routes/modules/` 当前为空属于设计结果，不表示缺页面；主路由来源是后端菜单
- 表单和 codegen 只能使用 `src/adapter/component/index.ts` 已注册、已适配的组件
- 上传相关复用组件集中在 `src/components/file-manager`、`src/components/upload`、`src/components/tinymce`、`src/components/json-editor`

## 开发入口

推荐 clone 后优先使用根目录的 Docker 入口启动完整开发环境：

```powershell
.\docker\dev\compose.ps1 up -d --build
.\docker\dev\compose.ps1 down -v
.\docker\dev\compose.ps1 --profile frontend up -d --build frontend
```

```bash
./docker/dev/compose.sh up -d --build
./docker/dev/compose.sh down -v
./docker/dev/compose.sh --profile frontend up -d --build frontend
```

脚本会先把 `docker/dev/.env` 同步到 `admin-go/.env`，再调用对应的 `docker compose`。
默认启动不会拉起前端；前端需要显式启用 `frontend` profile，避免低配机器启动时自动执行 `pnpm install`。

### 本地直跑入口

需要直接调试代码时，当前入口如下：

```bash
cd admin-go
go run ./app/system/main.go
go run ./app/upload/main.go

cd ../vue-vben-admin
pnpm dev:antd
```

后端通常分两个终端分别启动 `system` 和 `upload`；前端默认读取 `apps/web-antd/.env.development`，通过 `/api` 代理到这两个后端服务。

### 初始化数据

- 初始化与结构变更统一走 `admin-go/database/migrations/`
- Docker / 部署启动前必须先执行 `golang-migrate up`
- 默认上传配置使用本地存储，目录为 `resource/upload`

## 重要约定

- 不要把 `dao/`、`model/do/`、`model/entity/` 当业务层维护
- 菜单删改要同时检查前端入口和 `system_menu` 初始化数据
- 后端菜单里的 `path`、`component`、`type`、`linkURL/linkUrl` 必须和 `web-antd` 路由契约保持一致
- 数据库结构变更、初始化数据变更必须新增 `golang-migrate` 迁移文件，禁止继续维护旧 `init.sql` 作为真源
- 管理端前端和 `admin-go/codegen` 生成模板，必须服从 `vue-vben-admin/apps/web-antd/src/adapter/component/index.ts` 的组件注册规则
- 如果生成需要的新组件不在 adapter 中，必须先在 adapter 完成适配，再改生成器；禁止在模板里直接使用仓库未注册的组件名
- 生成页的页面组织和交互方式，优先对齐 `views/system/*`、`views/upload/*` 的现有 `vben` 写法
- 统一响应结构仍为 `{ code, message, data }`
- ID 继续使用雪花 ID / 字符串化 JSON 约定
- 默认初始化账号为 `admin` / `admin123`
- 仓库根目录 `lefthook.yml` 已启用：提交前会校验页面规范，推送前会补跑后端 / codegen / 前端核心校验
