# GBaseAdmin 代码库导读

这份文档基于当前仓库代码整理，目的不是替代各子目录 README，而是补一份从仓库根目录出发的“代码地图”和“维护手册”。

## 仓库定位

当前仓库是一个收缩后的后台基础框架，长期维护的核心范围只有四块：

- `admin-go/app/system`：登录、用户、角色、部门、菜单、权限
- `admin-go/app/upload`：上传配置、目录、目录规则、文件记录、文件上传
- `vue-vben-admin/apps/web-antd/src`：管理端前端，仅保留 system / upload 对应页面和公共壳
- `admin-go/codegen`：后端/前端/菜单代码生成器

几个关键真源要先分清：

| 主题 | 真源 |
| --- | --- |
| 数据库结构、基线数据 | `admin-go/database/migrations/` |
| 后端路由挂载 | `admin-go/app/*/internal/cmd/cmd.go` |
| 前端动态菜单契约 | `admin-go/app/system/internal/logic/auth/auth.go` + `vue-vben-admin/apps/web-antd/src/api/core/menu-transformer.ts` |
| 生成页可用组件清单 | `vue-vben-admin/apps/web-antd/src/adapter/component/index.ts` + `admin-go/codegen/parser/component_registry.go` |
| 仓库保留范围与菜单边界 | `contracts/baseadmin-scope.json` + `scripts/verify-baseadmin-scope.sh` |
| 低资源服务器重任务入口 | `scripts/run-go-task-with-limits.sh` + `scripts/run-node-task-with-limits.sh` |

## 后端架构

### 启动链路

两个应用的启动结构完全一致：

1. `main.go` 通过匿名导入注册 `logic`、打包资源和 MySQL / Redis 驱动
2. `internal/cmd/cmd.go` 创建 GoFrame Server 并绑定路由
3. `controller` 只做参数接入与返回结构封装
4. `service` 定义接口
5. `logic` 承担真正业务逻辑
6. `dao` / `model/entity` / `model/do` 是数据库访问层，不作为业务层维护

这条链路是整个后端的主干。后续新增模块时，优先保持这个分层，不要把业务直接堆进 controller 或 dao。

### system 应用

入口与路由：

- 启动入口：`admin-go/app/system/main.go`
- 路由绑定：`admin-go/app/system/internal/cmd/cmd.go`
- 公开路由：`GET /healthz`、`GET /readyz`、`GET /hello`
- 业务前缀：`/api/system`

当前挂载控制器：

- `auth`
- `dept`
- `role`
- `menu`
- `users`
- `health`
- `hello`

认证主链路：

1. 登录走 `POST /api/system/auth/login`
2. `logic/auth` 校验用户名、密码、账号状态，并做登录失败限流
3. 登录成功后生成 JWT
4. 已登录应用如果要跳转别的应用，可以调用 `POST /api/system/auth/ticket` 为当前用户生成短效签名票据
5. 目标应用收到票据后调用 `POST /api/system/auth/ticket-login` 换取自己的 JWT
6. `GET /api/system/auth/info` 返回当前用户信息、角色和权限码
7. `GET /api/system/auth/menus` 返回当前用户菜单树，供前端动态路由使用

`logic/auth` 还有几个值得记住的细节：

- 登录失败 5 次会触发 10 分钟限流
- `info` 和 `menus` 有 1 分钟缓存
- 应用间票据默认 60 秒过期，并通过 Redis 做一次性重放保护
- `middleware/auth.go` 已经接了服务端权限校验，不再只靠前端按钮控制
- 超级管理员角色会跳过权限过滤
- 菜单权限来自 `system_role_menu`
- 数据权限来自 `system_role_dept`

健康检查也已经拆成两类：

- `GET /healthz`：进程存活
- `GET /readyz`：数据库可用性和关键依赖就绪检查

### upload 应用

入口与路由：

- 启动入口：`admin-go/app/upload/main.go`
- 路由绑定：`admin-go/app/upload/internal/cmd/cmd.go`
- 公开路由：`GET /healthz`、`GET /readyz`
- 业务前缀：`/api/upload`

当前挂载控制器：

- `config`
- `dir`
- `dir_rule`
- `file`
- `health`

上传主链路集中在 `admin-go/app/upload/internal/logic/uploader/uploader.go`：

1. 读取 `file` 和可选的 `dirId`
2. 查询默认上传配置 `upload_config`
3. 校验大小限制与目录存在性
4. 先把文件落到本地临时目录
5. 根据 `storage` 决定保留本地、上传 OSS 或上传 COS
6. 写入 `upload_file`
7. 数据库写入失败时回滚本地文件或云端对象

这条实现说明了一件事：上传模块以数据库记录为最终一致性中心，不是单纯“把文件写磁盘”。

当前上传侧还有两个维护边界要记住：

- 配置删除会拦截仍被云文件引用的 OSS/COS 配置
- 文件删除现在以数据库记录一致性为主线：本地文件先原子暂存再删记录，云文件删失败会恢复对应记录

### utility 层

`admin-go/utility` 是被两个应用复用的基础能力，当前较重要的几个包：

- `jwt`：Token 生成与解析
- `password`：bcrypt 封装
- `snowflake`：雪花 ID 和 JSON 安全类型
- `treeutil`：树结构组装
- `cache`：缓存封装
- `pageutil`：分页辅助
- `response`：统一响应结构支持

业务逻辑里如果出现“每个模块都在重复写”的通用能力，优先考虑落到这里。

## 前端架构

### 当前前端范围

实际维护的是 `vue-vben-admin/apps/web-antd/src`，其中关键目录如下：

- `api/core`：登录、用户信息、菜单转换等核心接口
- `api/system`：system 业务 API
- `api/upload`：upload 业务 API
- `adapter/component`：Vben 表单/弹窗/抽屉可用组件适配层
- `components/file-manager`：文件管理器
- `components/upload`：图片/文件上传组件
- `components/tinymce`：富文本编辑器
- `components/json-editor`：JSON 编辑器
- `store/auth.ts`：登录态主链路
- `views/system/*`、`views/upload/*`：当前保留业务页
- `views/_core/*`：认证页和 fallback 页

### 登录与权限

登录链路集中在 `vue-vben-admin/apps/web-antd/src/store/auth.ts`：

1. `loginApi()` 获取后端返回的 `token`
2. token 写入 access store
3. `getAuthInfoApi()` 获取当前用户信息
4. `extractAccessCodes()` 从 `perms` 提取权限码
5. 用户信息写入 user store
6. 跳转到默认首页或用户首页

这套前端不是本地硬编码角色/菜单模式，而是完全依赖后端返回的权限与菜单。

### 动态菜单与路由

动态菜单链路如下：

1. 前端调用 `GET /system/auth/menus`
2. `src/api/core/menu.ts` 读取后端菜单数组
3. `src/api/core/menu-transformer.ts` 把后端菜单转换成 Vben 路由

几个关键约定：

- 只保留 `type=1/2/4/5` 且 `isShow=1` 的菜单进入路由树
- `type=3` 按钮权限不会进入路由
- `type=4` 转成 `meta.link`
- `type=5` 转成 `meta.iframeSrc`
- 外链 / 内嵌页缺少 `component` 时前端会兜底成 `IFrameView`
- 前后端现在都会拦截坏菜单：路由菜单必须有合法 `path`，`type=2` 必须有 `component`，`type=4/5` 必须有 `linkURL`，重复 `path` 会被拒绝或跳过

### 角色与数据权限边界

- 用户表单拿到的角色树不再是“全部角色”，而是当前操作者可分配的启用角色子集
- 服务端对用户创建、更新、删号、重置密码都会复查目标用户当前启用角色，避免通过操作高权限非超管账号完成提权
- 角色数据权限里的自定义部门授权，后端会再次按当前操作者的数据范围过滤，不能通过手工构造请求越权授予部门

因此，改菜单时不要只看前端页面文件，还要同时检查后端菜单数据和菜单转换逻辑。

### 业务页与组件边界

当前页面只保留：

- `views/system/dept/index.vue`
- `views/system/menu/index.vue`
- `views/system/role/index.vue`
- `views/system/users/index.vue`
- `views/upload/config/index.vue`
- `views/upload/dir/index.vue`
- `views/upload/dir_rule/index.vue`
- `views/upload/file/index.vue`

这些列表页现在都已经统一支持：

- 复选框选择
- 批量删除
- 对应 `/batch-delete` 后端接口

`router/routes/modules/` 为空是当前设计结果，不代表缺页面。业务入口主要来自后端动态菜单。

组件边界里最重要的是 `adapter/component/index.ts`：

- 它是 Vben 表单组件适配层
- 也是 codegen 允许生成哪些组件的前端边界
- 新增表单组件时，必须先在这里完成适配，再去改 codegen 映射

### 上传复用链路

上传模块的前后端复用边界很明确：

- 文件上传 API：`src/api/upload/file/index.ts`
- 文件管理器弹窗：`src/components/file-manager/use-file-manager.ts`
- 表单组件：`ImageUpload`、`FileUpload`
- 富文本图片上传：`components/tinymce`

`components/file-manager` 现在不只支持“多选已有文件”，在 `multiple=true` 场景下也支持一次选择多个本地文件并发上传。

也就是说，上传能力不是只服务 `views/upload/file` 页面，而是作为全局基础组件向其他业务页输出。

## 代码生成器架构

`admin-go/codegen` 是一个完整子系统，不是简单脚本集合。

### 目录职责

- `main.go`：CLI 入口，解析参数，组织生成流程
- `parser/`：解析表结构、字段规则、外键关系、组件映射
- `generator/backend`：生成后端文件
- `generator/frontend`：生成前端文件
- `generator/menu`：生成菜单计划和菜单写库
- `generator/util`：模板缓存、落盘、计划提交等工具
- `templates/backend`：后端模板
- `templates/frontend`：前端模板
- `cmd/verifye2e`：端到端验证入口
- `testdata/golden`：快照真源

### 生成主流程

`main.go` 当前的核心流程是：

1. 读取 CLI 参数和 `codegen.yaml`
2. 用 parser 解析一张或多张表
3. 按 `only` 决定生成 backend、frontend、menu
4. 先产出文件计划，再统一落盘
5. 需要时写出 manifest
6. 按开关决定是否执行 `gf gen dao`、`gf init`

当前生成器对低资源机器已经做了取舍：

- `--with-dao` 默认关闭
- `--with-init` 默认关闭
- 菜单写库支持批量事务
- 列表页模板默认带复选框和批量删除接口
- `dry-run` 和 manifest 已经能用于审查和 CI

### 维护时最重要的契约

- 表名必须满足 `{app}_{module}`
- 组件名必须落在前端 adapter 和 codegen 注册表共同支持的范围
- 数据库注释会直接影响前端 label、枚举、Tooltip、搜索方式和组件映射
- 结构变更先改迁移，再谈生成器

## 数据库与迁移

数据库真源只有 `admin-go/database/migrations/`。

当前迁移文件包括：

- `000001_baseline_system_upload`
- `000002_fill_missing_menu_icons`

维护规则：

- 新增表、改字段、补默认菜单、补默认账号、补上传默认配置，都必须新增迁移
- 不再把 `docker/mysql/init.sql` 或 `codegen/sql/*.sql` 当真源
- Docker 启动前要先跑 `migrate up`

## 开发、校验与部署

### Docker 开发

根目录开发入口集中在 `docker/dev/`：

- `compose.sh`
- `compose.ps1`
- `docker-compose.yml`
- `docker-compose.cn.yml`
- `.env`

`docker/dev/compose.sh` 做了两件重要的事情：

- 每次启动前把 `docker/dev/.env` 同步覆盖到 `admin-go/.env`
- 只有显式请求 `frontend` profile 时才启动前端

前端被单独做成 profile 的原因是：低配机器上 `pnpm install` 和 `pnpm build` 很容易打爆内存。

### 低资源服务器约束

当前仓库已经补了受限执行入口：

- `scripts/run-go-task-with-limits.sh`
- `scripts/run-node-task-with-limits.sh`
- `scripts/wait-for-cpu-idle.sh`

它们会在任务启动前先等 CPU 回落，再以保守的 CPU、内存和并发限制执行；运行过程中如果宿主机再次冲到阈值上方，会自动暂停并在回落到恢复阈值后继续。手工跑 `go test`、`pnpm typecheck`、`pnpm test` 这类重任务时，优先走这些脚本而不是裸跑。

### 验证入口

- 默认验证路径是：`git commit` -> `git push` -> GitHub Actions
- 日常验证看 `CI GBaseAdmin`
- `main` 分支推送除了 `CI GBaseAdmin`，还会继续进入 `Deploy GBaseAdmin`
- `CI GBaseAdmin` 会输出 Summary，并上传 JUnit、JSON、文本日志 artifact 作为测试记录
- `Deploy GBaseAdmin` 会用专用 smoke 账号做真实目录规则上传冒烟，不再拿管理员账号跑自动化
- 根目录 `lefthook.yml` 只保留提醒，不再本地跑仓库级重校验

如需手工排查，再执行下面这组命令：

```bash
# 仓库范围与页面规范
bash ./scripts/verify-baseadmin-scope.sh
bash ./scripts/verify-vben-pages.sh

# 后端与 codegen 测试（带负载守卫）
cd admin-go && ../scripts/run-go-task-with-limits.sh go test ./...
cd admin-go/codegen && ../../scripts/run-go-task-with-limits.sh go test ./...

# 前端校验（带负载守卫）
bash ./scripts/run-node-task-with-limits.sh pnpm -C vue-vben-admin -F @vben/web-antd typecheck
bash ./scripts/run-node-task-with-limits.sh pnpm -C vue-vben-admin test:web-antd
```

其中 `scripts/verify-vben-pages.sh` 专门约束管理端页面风格，防止页面写法偏离当前仓库的 Vben 适配方案。
`scripts/verify-baseadmin-scope.sh` 则专门约束当前仓库只保留 `system/upload` 应用和菜单范围。

如果需要检查真实数据库是否已经被初始化成核心范围，可以执行：

```bash
bash ./scripts/check-db-core-scope.sh
```

### GitHub Actions

`.github/workflows/ci.yml` 和 `.github/workflows/deploy.yml` 分工如下：

- `ci.yml`：按路径判断 backend / frontend / infra 变更并分流校验
- `ci.yml`：执行后端测试、codegen 测试、范围校验、页面规则校验、前端 typecheck、`web-antd` 单测
- `ci.yml`：把后端/前端测试结果写进 Actions Summary，并上传 JUnit、JSON、文本日志 artifact
- `deploy.yml`：`main` 分支在部署前再次做上线门禁，再顺序部署 `system`、`upload`、`frontend`
- `deploy.yml`：自动确保专用 smoke 角色/账号存在，执行历史 `upload_file.dir_id` 回填，并在部署后跑真实目录规则上传冒烟

这里的 `infra` 指 `.github/workflows/**`、`scripts/`、`contracts/`、`lefthook.yml`。这类基础设施改动会触发校验，但不会在 `auto` 模式下误触发应用部署。

后端部署是逐个服务替换二进制并做健康检查；前端部署是打包 `dist` 后上传到服务器 release 目录，再切换符号链接。部署阶段需要仓库 secrets：`DEPLOY_SSH_KEY`、`SMOKE_USERNAME`、`SMOKE_PASSWORD`。

## 日常维护红线

下面这些改动最容易出“只改一处，线上另一处炸掉”的问题：

- 改数据库结构：同时检查迁移、后端 DTO、前端类型、codegen 模板
- 改菜单字段或菜单类型：同时检查 `system_menu` 数据、`auth/menus`、`menu-transformer.ts`
- 新增表单组件：同时检查 `adapter/component/index.ts` 和 `codegen/parser/component_registry.go`
- 改上传逻辑：同时检查 upload 后端逻辑、文件管理器、富文本上传
- 改前端启动或构建脚本：同时检查 Docker 启动脚本、CI、低资源限制脚本
- 改 Docker 编排：同时检查 `docker/dev/docker-compose.yml`、`docker/dev/docker-compose.cn.yml`、`docker/prod/docker-compose.yml`
