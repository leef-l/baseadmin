# 持续优化执行说明

用于长期执行“巡检 -> 修复 -> 回归 -> 发布”的重复硬化流程，避免人工轮转时遗漏重型校验和发布节奏。

## 执行入口

- 单轮执行：`./scripts/run-hardening-campaign.sh --start 2 --rounds 1`
- 连续执行：`./scripts/run-hardening-campaign.sh --start 1 --rounds 50 --publish-every 50`

## 后台运行与日志

- 推荐用 `tmux` 跑长任务，避免终端断开后进程退出
- 建议先创建日志目录：`mkdir -p /tmp/baseadmin-hardening`
- 从第 4 轮接着跑到第 500 轮：

```bash
tmux new-session -d -s baseadmin500 \
'cd /www/wwwroot/project/baseadmin && ./scripts/run-hardening-campaign.sh --start 4 --rounds 497 --publish-every 50 2>&1 \
| tee /tmp/baseadmin-hardening/rounds-4-500.log'
```

- 从头重跑 500 轮：

```bash
tmux new-session -d -s baseadmin500 \
'cd /www/wwwroot/project/baseadmin && ./scripts/run-hardening-campaign.sh --start 1 --rounds 500 --publish-every 50 2>&1 \
| tee /tmp/baseadmin-hardening/rounds-1-500.log'
```

- 实时看日志：`tail -f /tmp/baseadmin-hardening/rounds-4-500.log`
- 回到会话：`tmux attach -t baseadmin500`
- 停止任务：`tmux kill-session -t baseadmin500`
- 如果机器没有 `tmux`，可退回 `nohup`，日志同样写到 `/tmp/baseadmin-hardening/*.log`

## 固定节奏

- 每轮都会打印 `第N轮开始`
- 每轮开始前都会先检查服务器负载；超过 `80%` 就等待，直到回落到 `50%` 以下再继续
- 每轮默认执行 scope 校验、页面规则校验、后端测试、codegen 测试、离线模板验证
- 前端默认每轮执行 `web-antd` typecheck 和单测；如需跳过，可临时设置 `HARDENING_SKIP_FRONTEND=1`
- 每到 `50` 的倍数轮次，会额外执行 `codegen verifye2e --stage all`
- 如果工作区存在 `admin-go/codegen/` 或前端组件适配层相关改动，脚本会额外触发一次 demo 应用全面测试；该测试在临时工作区执行，不把 `app/demo` 或前端 demo 产物落到当前仓库
- 命中上述 codegen 相关改动后，如未同步更新 `docs/流程日志/优化审计记录.md`，脚本会直接失败，禁止把本轮记为完成
- 每到 `50` 的倍数轮次，会更新本说明中的关口验证轮次，并在工作区有变更时自动 `git add/commit/push`

## 未满 50 轮时的补充要求

- 当前从第 `4` 轮继续推进时，尚未完成第 `50` 轮关口；在到达第 `50` 轮之前，不能把 `50` 轮节点当作已完成。
- 只要本轮改动涉及 `admin-go/codegen/`、模板、parser、generator、菜单生成或生成结果契约，不要等到第 `50` 轮才做重型回归，必须额外执行一次 demo 应用全面测试。
- demo 应用全面测试的目标不是只看离线快照，而是覆盖 `demo_category`、`demo_article`、`demo_tag` 这组三张演示表对应的真实生成链路，确认树表、外键、枚举、上传、富文本、搜索、批量操作等场景没有被模板修改打坏。
- demo 应用全面测试不是“跑一下命令就结束”。如果测试过程中发现生成错误、编译错误、类型错误、DAO 生成失败、菜单异常、导入导出异常或模板输出漂移，必须先修复问题，再重新执行完整验证，直到本轮相关检查全部通过后才能进入下一轮。
- 建议顺序：

```bash
cd /www/wwwroot/project/baseadmin/admin-go/codegen

# 1. 先做离线模板验证和单测
../../scripts/run-go-task-with-limits.sh go run verify_codegen.go
../../scripts/run-go-task-with-limits.sh go test ./...

# 2. 再做生成级端到端验证（verifydemo_* 临时表）
../../scripts/run-go-task-with-limits.sh go run ./cmd/verifye2e --stage all

# 3. 如本轮改动触及模板/字段映射/关联推断，再补 demo 应用真实生成验证
# 先确保目标 MySQL 已导入 admin-go/codegen/sql/demo.sql
../../scripts/run-go-task-with-limits.sh go run . --table demo_category,demo_article,demo_tag --with-init --with-dao --force
```

- 第 3 步完成后，至少还要补一轮真实输出验证：
  - 后端：`cd /www/wwwroot/project/baseadmin/admin-go && ../scripts/run-go-task-with-limits.sh go test ./app/demo/...`
  - 前端：`cd /www/wwwroot/project/baseadmin && ./scripts/run-node-task-with-limits.sh pnpm -C vue-vben-admin -F @vben/web-antd typecheck`
- 当前自动脚本会把这组 demo 验证放到临时工作区完成，并在结束后清理临时目录和 `demo_*` 验证表；如果目标库里已经存在 `demo_category`、`demo_article`、`demo_tag`，脚本会拒绝覆盖并直接失败
- 临时前端工作区的校验以 `apps/web-antd` 目录下直接执行 `vue-tsc --noEmit --skipLibCheck` 为准，不使用 `pnpm -F @vben/web-antd typecheck` 作为唯一判定，避免 `pnpm recursive run` 在复制工作区里的假失败影响本轮结论
- demo 应用全面测试和修复完成后，必须再回看本轮涉及的所有相关代码，不能只盯报错点。至少要重新检查：
  - `admin-go/codegen/` 下本轮涉及的模板、parser、generator、verify、脚本入口
  - 生成出来的 `app/demo` 后端代码和 `vue-vben-admin` 对应前端输出
  - 与本轮改动相连的公共层代码，例如菜单、适配层、校验脚本、资源限制脚本
- 代码回看完成后，必须同步更新文档，至少包含：
  - `docs/流程日志/优化审计记录.md`：记录本轮发现的问题、修复点、验证结果、剩余风险
  - `docs/流程日志/持续优化执行说明.md`：如流程、验证门槛、执行命令、关口说明发生变化，要及时修订
- 如果 demo 应用全面测试失败，或修复后未完成代码回看、文档更新，本轮不能记为完成，必须继续处理后再进入下一轮。

## 负载约束

- 所有 Go / Node 重任务都必须通过 `scripts/run-go-task-with-limits.sh` 或 `scripts/run-node-task-with-limits.sh`
- 所有 `pnpm` / `npm` / `npx` / `pnpx` 的重命令禁止裸跑，统一走 `scripts/run-node-task-with-limits.sh`
- 两个脚本都会先调用 `scripts/wait-for-cpu-idle.sh` 做启动前等待
- 默认 CPU 使用率达到 `80%` 就不再继续硬顶：
  - 启动前达到 `80%`：不启动，先等待
  - 运行中达到 `80%`：对整组任务发送 `STOP` 暂停，不杀进程、不丢现场
  - 回落到 `50%`：自动发送 `CONT` 恢复，从中断点继续执行
- 这个暂停/恢复逻辑由 `scripts/run-task-with-load-guard.sh` 统一提供，`run-go-task-with-limits.sh` 和 `run-node-task-with-limits.sh` 已接入
- Go / Node 两套包装脚本会分别输出 `[go-guard]`、`[node-guard]` 前缀，便于从长日志里区分是哪类任务触发了暂停/恢复
- 可通过 `RESOURCE_LOAD_MAX_PERCENT` / `RESOURCE_LOAD_RESUME_PERCENT` 调整阈值
- 2 核 4G 服务器上的常用写法：
  - `./scripts/run-node-task-with-limits.sh pnpm -C vue-vben-admin install --frozen-lockfile`
  - `./scripts/run-node-task-with-limits.sh pnpm -C vue-vben-admin -F @vben/web-antd typecheck`
  - `./scripts/run-node-task-with-limits.sh npm --prefix vue-vben-admin run build`
